# Skill: 系统初始化开发流程

> **适用范围**：Unity项目中新增业务系统（如地图系统、单位系统、战斗系统等）的完整开发流程
> **示例系统**：地图系统（Map System）
> **文档版本**：v1.1

---

```

│                     系统初始化开发流程                             │
├─────────────────────────────────────────────────────────────────┤
│  阶段1: 文档审查        →  阶段2: 系统设计                        │
```

## 2. 详细步骤与产出

### 阶段1: 文档审查（Read Phase）

**目标**：理解项目约束和技术栈规范

| 步骤 | 动作 | 产出 |
|------|------|------|
| 1.1 | 读取 `01-项目级文档/技术栈描述.md` | 理解架构分层、系统边界、扩展接口 |
| 1.2 | 读取 `01-项目级文档/项目级文档约束.md` | 明确服从义务、查阅要求 |
| 1.3 | 读取 `02-系统级文档/系统级文档约束.md` | 理解系统文档组织方式 |
| 1.4 | 读取 `03-对象级文档/对象级文档约束.md` | 理解类文档组织方式 |
| 1.5 | 检查目标系统的技术栈章节 | 提取功能边界和扩展接口 |

**关键产出**：
- 功能边界清单
- 必须实现的接口列表
- 技术选型约束（如六边形网格、ScriptableObject等）

---

### 阶段2: 系统设计（Design Phase）

**目标**：产出完整的系统文档（系统级 + 对象级）

#### 2.1 系统级文档创建

**询问用户确认**：
- 系统文件夹路径：`02-系统级文档/{系统名称}/`
- 主文档文件名：`{系统名称}系统设计.md`

**文档必须包含**：
```markdown
1. 模块概述
   - 职责边界（职责项 + 边界外事项）
   - 设计目标（可扩展性、性能、数据驱动）

2. 结构设计
   - 组件关系图（Mermaid/文本图）
   - 类职责矩阵

3. 接口定义
   - 对外暴露接口（供其他系统调用）
   - 扩展接口（模组支持）

4. 依赖声明
   - 依赖的其他系统
   - 被依赖关系

5. 数据配置方案
   - JSON/ScriptableObject配置结构
   - 配置示例

6. 引用声明
   > 服从：01-项目级文档/技术栈描述.md
```

**产出**：`02-系统级文档/{系统名称}/{系统名称}系统设计.md`

---

#### 2.2 对象级文档创建

**询问用户确认**：
- 对象级文件夹路径：`03-对象级文档/{系统名称}/`
- 命名规范：`{类名}_{中文描述}.md`

**每个类文档必须包含**：
```markdown
1. 类职责 - 单一职责说明
2. 继承关系 - 父类、实现的接口
3. 属性定义 - 字段、属性及访问级别
4. 方法清单 - 公有/保护/私有方法
5. 封装设计 - 对外暴露 vs 内部实现
6. 依赖类 - 关联的其他类
7. 引用链
   > 服从：01-项目级文档/技术栈描述.md
   > 服从：02-系统级文档/{系统名称}/{系统名称}系统设计.md
```

**产出清单示例**（以地图系统为例）：
| 类名 | 文件名 | 职责 |
|------|--------|------|
| ITerrainEffect | `ITerrainEffect_地形效果接口.md` | 契约接口，定义地形效果标准 |
| HexCoord | `HexCoord_六边形坐标系统.md` | 六边形坐标表示与计算 |
| HexTile | `HexTile_六边形地块.md` | 地块数据存储与单位引用 |
| HexGrid | `HexGrid_六边形网格管理.md` | 网格生成与管理，实现数据接口 |

**产出**：`03-对象级文档/{系统名称}/{类名}_{中文描述}.md`

### 阶段3: 代码规划（Plan Phase）

**目标**：确定代码文件组织结构和创建路径

#### 3.1 询问用户确认代码目录

**必须询问的问题**：

| 问题 | 示例答案 | 说明 |
|------|----------|------|
| 模块根目录？ | `Assets/Scripts/Map/` | 系统代码根路径 |
| 接口存放位置？ | `Assets/Scripts/Map/API/` | 契约接口命名空间建议：`TBS.{系统}.API` |
| 组件存放位置？ | `Assets/Scripts/Map/Components/` | MonoBehaviour组件 |
| 数据类存放位置？ | `Assets/Scripts/Map/Data/` | ScriptableObject/数据类 |
| 工具类存放位置？ | `Assets/Scripts/Map/Tools/` | 工具struct/静态类 |
| Prefab存放位置？ | `Assets/Prefabs/Map/` | 注意单复数（Prefabs vs Prefab） |

**目录结构确认模板**：
```
Assets/
├── Scripts/
│   └── {系统名}/
│       ├── API/          # 契约接口
│       ├── Components/   # MonoBehaviour组件
│       ├── Data/         # 数据类
│       └── Tools/        # 工具类
└── Prefabs/
    └── {系统名}/         # 预制体
```

**产出**：经用户确认的代码目录结构

---

### 阶段4: 代码实现（Implement Phase）

**目标**：按文档实现所有代码文件

**⚠️ 关键纪律：每个文件创建前必须询问路径确认**

#### 4.0 文件创建流程（严格遵循）

```
列出待创建文件清单
        ↓
询问用户确认每个文件的路径
        ↓
等待用户明确回复（是/修改路径）
        ↓
按确认的路径逐个创建
        ↓
创建完成后汇报
```

**禁止行为**：
- ❌ 未经确认直接创建文件
- ❌ 先创建后询问
- ❌ 批量创建后统一汇报

**正确示例**：
```
助手：接下来创建代码文件，请确认以下路径：

1. IMapDataProvider.cs → Assets/Scripts/Map/API/
2. HexGrid.cs → Assets/Scripts/Map/Components/
3. HexTile.cs → Assets/Scripts/Map/Components/

请确认或修改路径。

用户：确认，但 HexTile.cs 放到 UI/ 目录
助手：收到，HexTile.cs 改为放到 Assets/Scripts/Map/UI/
[然后才开始创建]
```

#### 4.1 实现顺序建议

```
1. 契约接口（API/）      → 2. 数据类（Data/）
         ↓                     ↓
3. 工具类（Tools/）      → 4. 组件类（Components/）
```

#### 4.2 代码实现规范

**每个文件必须包含**：
- 命名空间声明（按目录层级）
- XML文档注释（///）
- 区域化组织（#region）
- 依赖引用（using）

**命名规范**：
```csharp
// 命名空间
namespace TBS.{系统}.{分类}

// 接口（契约层）
public interface I{功能}Effect { }

// MonoBehaviour组件
public class {功能}Component : MonoBehaviour, I{接口} { }

// ScriptableObject数据
public class {功能}Data : ScriptableObject, I{接口} { }

// 工具struct
public readonly struct {功能}Coord : IEquatable<{功能}Coord> { }
```

#### 4.3 文件清单模板

| 序号 | 文件名 | 建议路径 | 类型 | 关键实现 |
|------|--------|----------|------|----------|
| 1 | `I{功能}Effect.cs` | `API/` | 接口 | 契约方法定义 |
| 2 | `{功能}Data.cs` | `Data/` | ScriptableObject | 数据存储+接口实现 |
| 3 | `{功能}Coord.cs` | `Tools/` | struct | 坐标计算+运算符重载 |
| 4 | `{功能}Tile.cs` | `Components/` | MonoBehaviour | 单位管理+事件 |
| 5 | `{功能}Grid.cs` | `Components/` | MonoBehaviour | 网格生成+查询接口 |

**⚠️ 注意**：以上路径仅为**建议**，必须经用户确认后方可创建。

**产出**：所有.cs代码文件

## 3. 本次示例（地图系统）完整产出

### 3.1 文档产出

| 层级 | 路径 | 文件数 |
|------|------|--------|
| 系统级 | `.cursor/02-系统级文档/地图系统/` | 1 |
| 对象级 | `.cursor/03-对象级文档/地图系统/` | 4 |
| 代码 | `Assets/Scripts/Map/` | 6 |

### 3.2 代码产出

| 分类 | 文件 | 职责 |
|------|------|------|
| API | `ITerrainEffect.cs` | 地形效果契约接口 |
| Data | `TerrainData.cs` | 地形数据ScriptableObject |
| Tools | `HexCoord.cs` | 六边形坐标struct |
| Tools | `MapConfigLoader.cs` | JSON配置加载器 |
| Components | `HexTile.cs` | 地块MonoBehaviour |
| Components | `HexGrid.cs` | 网格管理器 |

### 3.3 关键设计决策

1. **坐标系统**：采用轴向坐标（Axial Coordinates）为主，立方体坐标用于计算
2. **数据驱动**：地形通过ScriptableObject配置，地图通过JSON加载
3. **接口分层**：契约接口（API）+ 默认实现（Data/Components）
4. **扩展预留**：ITerrainEffect支持模组扩展自定义地形效果

## 5. 注意事项

### 5.1 常见陷阱

| 陷阱 | 解决方案 |
|------|----------|
| **未经确认创建文件（严重）** | 每个文件创建前必须列出清单并等待用户确认 |
| 代码目录与命名空间不匹配 | 严格按照目录层级定义namespace |
| 缺少XML文档注释 | 所有public成员必须加///注释 |
| 契约接口与实现分离不清 | API目录只放接口，不放实现 |
| 对象级文档引用链不完整 | 必须引用技术栈描述和系统设计文档 |

### 5.2 违规示例（反面教材）

**违规场景**：地图系统测试文件创建

| 文件 | 未经确认创建的路径 | 用户要求的路径 | 处理方式 |
|------|-------------------|---------------|----------|
| MapSystemTest.cs | `Scripts/Map/`（擅自决定） | `Scripts/Map/Test/` | 删除原文件，按用户路径重建 |
| MapTestEditor.cs | `Scripts/Map/Editor/`（擅自决定） | 保留原路径 | 本次接受，下次必须确认 |
| HexTileRenderer.cs | `Scripts/Map/Components/`（擅自决定） | 保留原路径 | 本次接受，下次必须确认 |

**用户反馈**："你创建新的文件时，没有询问我路径，这个非常不好"

**教训**：即使看似合理的路径，也必须在创建前明确询问用户确认。

### 5.3 质量保证点

1. **路径确认**：每个文件创建前必须获得用户明确确认
2. **编译检查**：代码创建后询问是否需要检查编译错误
3. **命名空间一致性**：`TBS.{系统}.{分类}` 必须与目录匹配
4. **事件设计**：MonoBehaviour组件应提供Unity事件和C#事件两种
5. **扩展预留**：接口设计应考虑模组扩展需求
